Field notes on agent infrastructure

The fastest Google Workspace CLI wasn't the one my agents preferred.

A brittle Gmail connector sent me looking for something long-running agents could discover, recover, and use across multiple Google identities without restarting.

Published July 2026 6-minute read Apple Silicon benchmark

I needed an agent to access two different Gmail accounts. For most people, that means one work account and one personal account. Here I call them UmbrellaCorp and Weyland. The goal was simple: work with both mailboxes without confusing identities or depending on a Gmail connector that could disappear after a small authentication or configuration problem.

An agent may now run for hours or days. Restarting it because a static integration failed—or because a new tool was added—throws away momentum and context. A normal CLI is refreshingly boring: it can be installed, authenticated, inspected, and invoked at any point in the agent's lifetime.

The initial plan was one CLI per account. The benchmark changed the architecture.

The connector lifecycle no longer fits the agent lifecycle

My starting point was the Gmail MCP or built-in connector. When it worked, it was convenient. The problem was fragility: an authentication issue or connector failure could knock mail access out for the rest of an agent run.

In the coding-agent environments I tested, MCP configuration and tool definitions were loaded at startup. Adding a server, changing its configuration, or recovering from certain failures meant restarting the agent. That assumption made sense for short sessions; it does not fit agents that work continuously for hours or days.

Multiple accounts made the mismatch worse. A generic “Gmail” tool does not always make the active identity obvious. I wanted the identity in every command:

gog --account umbrella gmail search 'has:attachment'
gog --account weyland  gmail search 'has:attachment'

A CLI has an independent lifecycle. It can be installed after the agent starts, authenticated without rebuilding the session, retried after failure, and selected explicitly per invocation. The agent keeps its context while the integration is repaired underneath it.

Two CLIs, two different philosophies

I tested gws and gog. Both speak to Google Workspace APIs and both produce structured output, but they expose very different interfaces.

API-shaped

gws

A compact Rust binary whose commands closely mirror Google's resource and method hierarchy. It is fast, direct, and predictable if you already think in Google API paths.

 Task-shaped

gog

A broader Go CLI with human-scale verbs, native account aliases, machine-readable schemas, and unusually thoughtful safety controls for automation.

gws exposes systematic API paths such as gmail users messages attachments get. gog expresses the same intent as gmail attachment. The first is excellent when you think in Google API resources; the second asks less of an autonomous agent.

The benchmark

The tests ran on an Apple Silicon Mac using hyperfine. I compared gws 0.22.5 with gog 0.34.0. Startup used 50 runs after warmup. The live Gmail test used 15 runs per tool against the same users.getProfile API method. The calls used different Workspace tenants, so this is a practical comparison rather than a laboratory network benchmark.

gws won the systems benchmark: almost ten times faster to start, about one-fifth the startup memory, and a smaller binary. Then the network got involved. A real Gmail operation took roughly four-tenths of a second with either tool; the 15 ms difference was small enough to disappear into ordinary internet variation.

Rust won the stopwatch. It did not win the workflow.

What an agent actually prefers

Agents care about latency, but they also care about the number of decisions required to express an action safely. On that measure, gog was the stronger interface.

Agent concernBetter fitWhy
Raw startup and memorygwsTiny process overhead
Common mailbox tasksgogHigh-level search, reply, forward, and attachment verbs
Multiple identitiesgogNamed OAuth clients, domain mappings, and account aliases
Safety boundariesgogRead-only, no-send, allowlists, denylists, and dry runs
Arbitrary API coveragegwsDirect mapping to Workspace API resources
Structured discoverygogMachine-readable schemas and a generic Discovery API escape hatch

The decisive features were not performance features. gog offers read-only and no-send modes, command allowlists and denylists, dry runs, non-interactive operation, untrusted-content wrapping, and output selection. Those controls constrain an agent before it touches a mailbox and reduce the policy that must live in a prompt. That is a bigger operational win than saving 30 milliseconds at process startup.

One CLI, two accounts, no identity confusion

The final setup used gog for both accounts while keeping their credentials and browser sessions separate.

UmbrellaCorp OAuth client: umbrella
Alias: umbrella
Firefox container: UmbrellaCorp
+
Weyland OAuth client: weyland
Alias: weyland
Firefox container: Weyland

At a high level, each identity gets its own Google Cloud OAuth client, stored credentials, domain mapping, and short account alias. OAuth is completed in a matching Firefox container so existing Google cookies cannot silently select the wrong account. Secrets and refresh tokens live in the macOS Keychain. Finally, a harmless Gmail profile call verifies that each alias resolves to the intended mailbox.

You do not need to assemble this by hand. Codex can install the CLI, create and configure the Google Cloud projects, guide each OAuth flow in the correct browser container, register the aliases, and verify both accounts for real. The human only needs to approve Google's consent screens and any security-sensitive steps.

The practical verdict

Choose gws when process overhead, low memory, and a direct mapping to Workspace APIs dominate. Choose gog when autonomous agents will perform everyday work across multiple accounts and you want compact commands plus enforceable guardrails.

For this setup, the benchmark overturned the original assumption. The smaller, faster binary was technically impressive. The heavier CLI was the better agent tool. So both fictional accounts moved to gog, with explicit aliases, separate OAuth clients, separate browser containers, and the old CLI retained only as a rollback path.
