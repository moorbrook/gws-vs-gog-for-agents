Field notes on agent infrastructure

The fastest Google Workspace CLI wasn't the one my agents preferred.

A brittle Gmail connector sent me looking for something long-running agents could discover, recover, and use across multiple work identities without restarting.

Published July 2026 12-minute read Apple Silicon benchmark

I have more than one job, which means I also have more than one Google Workspace account. Let's call the companies UmbrellaCorp and Weyland. I wanted long-running coding agents to work with both mailboxes without confusing identities, leaking credentials across employers, or depending on a Gmail connector that could disappear from service after a small configuration or authentication problem.

That last constraint matters more now than it did a year ago. An agent may run for hours or days. Restarting it because a static integration failed—or because a new tool was added—is not a serious production workflow. A normal CLI is refreshingly boring: it can be installed, authenticated, upgraded, inspected, and invoked at any point in the agent's lifetime.

The initial plan was one CLI per company. The benchmark changed the architecture.

The connector lifecycle no longer fits the agent lifecycle

My starting point was the Gmail MCP or built-in Gmail connector. When it worked, it was convenient. The problem was operational fragility: an authentication issue, configuration mismatch, or connector failure could knock mail access out for the rest of an agent run.

In the coding-agent environments I tested, MCP configuration and tool definitions were loaded when the agent started. Adding a server, changing its configuration, or recovering from certain failures meant restarting the agent so it could rebuild its tool inventory. That assumption was tolerable when sessions lasted a few minutes. It makes much less sense when an agent may accumulate context and work continuously for hours or days.

Multiple accounts made the mismatch worse. A generic “Gmail” tool does not necessarily make the active identity obvious at every call. With two employers, silent account selection is not a convenience problem—it is a boundary problem. I wanted the identity in the command itself:

gog --account umbrella gmail search 'has:attachment'
gog --account weyland  gmail search 'has:attachment'

A CLI has an independent lifecycle. It can be installed after the agent starts, authenticated without rebuilding the session, inspected with --help or a schema command, retried after failure, and selected explicitly per invocation. The agent keeps its context while the integration is repaired underneath it.

Two CLIs, two different philosophies

I tested gws and gog. Both speak to Google Workspace APIs and both produce structured output, but they expose very different interfaces.

API-shaped

gws

A compact Rust binary whose commands closely mirror Google's resource and method hierarchy. It is fast, direct, and predictable if you already think in Google API paths.

Task-shaped

gog

A broader Go CLI with human-scale verbs, native account aliases, machine-readable schemas, and unusually thoughtful safety controls for automation.

The API-shaped design makes gws wonderfully systematic:

gws gmail users messages attachments get \
  --params '{"userId":"me","messageId":"…","id":"…"}' \
  --output invoice.pdf

The task-shaped design makes the equivalent intent shorter in gog:

gog --account umbrella gmail attachment \
  <message-id> <attachment-id> --out invoice.pdf

The benchmark

The tests ran on an Apple Silicon Mac using hyperfine. I compared gws 0.22.5 with gog 0.34.0. Startup used 50 runs after warmup. The live Gmail test used 15 runs per tool against the same users.getProfile API method. The calls used different Workspace tenants, so this is a practical comparison rather than a laboratory network benchmark.

Rust won the stopwatch. It did not win the workflow.

What an agent actually prefers

Agents care about latency, but they also care about the number of decisions required to express an action safely. On that measure, gog was the stronger interface.

Agent concernBetter fitWhy
Raw startup and memorygwsTiny process overhead
Common mailbox tasksgogHigh-level search, reply, forward, and attachment verbs
Multiple identitiesgogNamed OAuth clients, domain mappings, and account aliases
Safety boundariesgogRead-only, no-send, allowlists, denylists, and dry runs
Arbitrary API coveragegwsDirect mapping to Workspace API resources
Structured discoverygogMachine-readable schemas and a generic Discovery API escape hatch

The decisive flags were not performance flags:

--readonly
--gmail-no-send
--enable-commands / --disable-commands
--dry-run
--no-input
--wrap-untrusted
--results-only / --select

Those controls let an operator constrain an agent before it touches a mailbox. They also reduce the amount of policy that must live in a prompt. That is a much bigger operational win than saving 30 milliseconds when a process starts.

One CLI, two companies, no identity confusion

The final setup used gog for both fictional employers, while keeping credentials and browser sessions separate.

Each company gets its own Google Cloud project and Desktop OAuth client. The client secret goes into the macOS Keychain, not a shell profile. A domain mapping tells gog which OAuth client belongs to which account:

# Import each company's Desktop OAuth JSON under a distinct client.
gog --client umbrella auth credentials set umbrella-client.json \
  --domain umbrellacorp.example

gog --client weyland auth credentials set weyland-client.json \
  --domain weyland.example

# Authorize each account. Use its matching Firefox container.
gog --client umbrella auth add operator@umbrellacorp.example \
  --services gmail --gmail-scope full --manual

gog --client weyland auth add operator@weyland.example \
  --services gmail --gmail-scope full --manual

# Give agents short, unambiguous selectors.
gog auth alias set umbrella operator@umbrellacorp.example
gog auth alias set weyland operator@weyland.example

The Firefox container rule

Before opening an OAuth URL, verify the tab's container badge—not just the avatar on the Google page. UmbrellaCorp authorization happens only in an UmbrellaCorp container tab; Weyland authorization happens only in a Weyland tab. This prevents Google's account chooser and existing cookies from silently selecting the wrong employer.

With --manual, Google eventually redirects to a temporary 127.0.0.1 address. Firefox may show “Unable to connect.” That is expected: copy the complete redirect URL from the address bar and paste it back into the waiting CLI. The authorization code is in the URL.

Verify before handing it to an agent

gog auth doctor --check --no-input

gog --account umbrella api call gmail v1 users.getProfile \
  --params '{"userId":"me"}' --json --results-only

gog --account weyland api call gmail v1 users.getProfile \
  --params '{"userId":"me"}' --json --results-only

The profile check is important. Successful OAuth is not enough; the final verification should prove that each alias resolves to the intended mailbox and OAuth client.

Why this beats a static connector for long-running agents

A CLI is not automatically safer or better than every connector. The advantage here is lifecycle independence. The agent can discover the tool after startup, invoke it as a subprocess, inspect its schema, and recover from ordinary failures without reinitializing its entire tool environment.

It is also easier to test. We can benchmark the binary, run an authentication doctor, constrain commands, validate an alias against a profile endpoint, and capture stable exit codes. Those are conventional operational primitives, and conventional primitives age well.

The practical verdict

Choose gws when process overhead, low memory, and a direct mapping to Workspace APIs dominate. Choose gog when autonomous agents will perform everyday work across multiple accounts and you want compact commands plus enforceable guardrails.

For this setup, the benchmark overturned the original assumption. The smaller, faster binary was technically impressive. The heavier CLI was the better agent tool. So both fictional companies moved to gog, with explicit aliases, separate OAuth clients, separate browser containers, and the old CLI retained only as a rollback path.
